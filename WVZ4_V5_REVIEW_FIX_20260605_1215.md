# WVZ4 v5 bool-storage / dirty-peek storage fold review fix 20260605_1215

## Reviewed scope

This review focused on the v5 changes that split logical signal ids from physical storage ids, plus U01 bool-storage and value-mask compression changes.

## Fixed issues

### 1. Dirty-peek storage folding must not depend on emit_only_on_change

Before this fix, dirty-peek alias de-duplication was mostly achieved through the storage track's last-value state. That worked for the default `emit_only_on_change=true`, but when `emit_only_on_change=false`, multiple logical aliases sharing the same `storage_id` could emit duplicate physical storage samples in the same cycle, causing the WVZ4 recorder to reject the cycle with `duplicate storage sample in one cycle`.

Fix:

- Added `TrackRuntimeState::dirty_peek_storage_sample_epoch`.
- Added `Tracer::dirty_peek_storage_sample_epoch_` and `begin_dirty_peek_storage_sample_epoch_()`.
- `sample_one_dirty_peek_leaf_changed()` now samples each physical dirty-peek storage at most once per `sample_dirty_peek_groups()` call, independent of `emit_only_on_change`.

### 2. Logical liveness and physical storage state were separated for dirty-peek aliases

Before this fix, `sample_one_dirty_peek_leaf_changed()` used `storage_id` to access `track_runtime_` and checked `runtime.alive` on that storage/canonical logical track. If the canonical logical track were invalidated while another logical alias remained live, the physical storage stream could be incorrectly suppressed.

Fix:

- Check `track_runtime_[leaf.track_id].alive` for logical path liveness.
- Use `track_runtime_[storage_id]` only as the physical storage last-value/de-dup state.

### 3. Reduced avoidable unordered_map rehashing

The new `dirty_peek_storage_by_key_` map is topology-build time only, but it can receive many entries for large designs. Rebuilding the map now reserves enough capacity before insertion.

`PathStableWvz4Recorder::build_layout()` also reserves `name_to_id` using the number of declared nodes.

## Notes

The following choices remain intentional:

- `CycleValueUpdate::signal_id` still carries a physical storage stream id in v5. The name is retained for local compatibility inside writer code, but the comment states the new semantics.
- WVZ4 v5 remains a breaking format change because `SIGT` now stores `signal_id, storage_id, node_id, type, bit_width, radix`.
- The dirty-peek storage map remains an `unordered_map` because it is used during topology construction, not cycle-hot sampling. The cycle-hot path uses vector-indexed runtime state and per-sample epochs.
